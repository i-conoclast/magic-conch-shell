# D. Planning — 사용 시나리오

---

## Scenario D1.1: 기상 직후 아침 브리핑 수신

**Given**: 사용자가 오전 7시를 아침 브리핑 시간으로 설정했다.
**When**: 오전 7시가 된다.
**Then**:
- 사용자의 채팅 앱에 브리핑 메시지가 도착
- 내용 구성: 오늘 태스크 3건, 어제 미완료 1건, 최근 이메일에서 감지된 이벤트 1건, 이번 주 우선순위, "오늘 가장 미룰까 고민 중인 것은?" 질문
- 사용자가 메시지를 읽고 응답 가능한 상태

---

## Scenario D1.2: 외부 시스템 다운일 때 브리핑

**Given**: 외부 태스크 시스템이 일시 다운 상태.
**When**: 아침 브리핑이 실행된다.
**Then**:
- 외부 시스템에서 읽지 못한 정보는 "연결 불가"로 표시
- 로컬에 있는 정보(어제 기록·최근 엔티티 변화 등)로 축약된 브리핑이라도 전달
- 브리핑 실행이 실패하지 않음

---

## Scenario D2.1: 플랜 확정 대화

**Given**: 아침 브리핑이 오늘 태스크 3건을 제안했다.
**When**: 사용자가 "첫 번째 것은 내일로 미루고, 여기에 블로그 초안 쓰기 추가"라고 응답.
**Then**:
- 시스템이 수정 반영: 원래 1번 태스크는 내일로 이동, "블로그 초안 쓰기"가 오늘 목록에 추가
- 업데이트된 목록이 다시 표시됨
- 사용자가 "확정"이라고 입력할 때까지 추가 수정 가능

---

## Scenario D2.2: 실수 확정 방지

**Given**: 사용자가 플랜 확정 대화 중이다.
**When**: 사용자가 "오케이"라고 응답한다 (확정 아님).
**Then**:
- "확정하시려면 '확정'이라고 명시적으로 입력해주세요"라고 안내
- 대화 상태는 유지됨

---

## Scenario D3.1: 저녁 회고 일괄 처리

**Given**: 하루 동안 엔티티 초안 2건, 제안 3건이 쌓였다.
**When**: 사용자가 저녁 회고를 시작한다.
**Then**:
- 회고 흐름 중 엔티티 초안 2건을 하나씩 승인·거절 결정
- 제안 3건 처리
- 오늘 확정 플랜 대비 실제 실행 확인
- 느낀 점 한 줄 옵션 (사용자가 건너뛰어도 됨)
- 전체 흐름 5분 이내 종료

---

## Scenario D3.2: 회고 건너뛰기

**Given**: 사용자가 저녁 회고를 건너뛰고 자러 갔다.
**When**: 다음날 아침이 된다.
**Then**:
- 대기 중이던 초안·제안은 그대로 유지됨
- 아침 브리핑에 "어제 처리 못 한 승인 대기 N건이 있어요" 안내
- 사용자가 원할 때 승인 인박스를 열어 처리 가능

---

## Scenario D4.1: 주간 리뷰

**Given**: 일요일 저녁 8시가 주간 리뷰 시간이다.
**When**: 시간이 되어 시스템이 주간 리뷰를 생성한다.
**Then**:
- 한 주간 확정 플랜 중 완료/미완료 비율
- 도메인별 활동량 변화 (예: 커리어 기록 급증)
- 한 주간 3번 이상 연결된 엔티티 강조
- 다음 주 우선순위 제안 1~2개
- 결과가 노트 파일로 저장되어 나중에 참조 가능

---

## Scenario D5.1: 확정 플랜이 외부 시스템 반영

**Given**: 아침 플랜 대화에서 오늘 태스크 3건을 확정했다.
**When**: 확정 명령이 처리된다.
**Then**:
- 외부 태스크 시스템에 3건이 자동 생성됨
- 각 태스크에 마감일·도메인 정보 포함
- 성공 개수가 사용자에게 확인 메시지로 전달됨

---

## Scenario D5.2: 외부 시스템 장애 복구

**Given**: 외부 시스템이 장애 중이라 플랜 반영이 큐에 쌓여 있다.
**When**: 외부 시스템이 복구된다.
**Then**:
- 자동 재시도로 큐에 쌓인 플랜이 순차 반영됨
- 반영 완료 시 사용자에게 알림 (원하는 경우)
- 상태 명령으로 큐 잔여·반영 완료 이력 조회 가능
