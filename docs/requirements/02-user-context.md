# 02. User Context — 누가, 언제, 어떻게 쓰나

**작성일**: 2026-04-19
**관련 Step**: 인터뷰 Step 2 (사용자 맥락)

## 2.1 사용자 프로필

**단일 사용자, 솔로 운영.**

- 커리어 전환을 준비하는 개인 (개발·ML 관련 직무)
- 라이프 5개 영역(커리어·신체건강·멘탈건강·관계·재무)에 걸친 OKR을 직접 운영 중
- 이미 몇 가지 생산성 도구를 조합해 쓰고 있음 (자세한 건 외부, 이 문서 범위 밖)
- 언어: **한국어·영어 혼용**. 기록·입력·출력 모두 두 언어가 섞여 나올 수 있음
- 기술적으로는 터미널·CLI에 익숙

**성향 주의사항**:
- 잔소리·누락 감지 알림을 싫어함 (침입성에 민감)
- 반복해서 같은 설명을 하는 것에 피로도 큼
- 데이터 근거 없는 답변을 불신함

## 2.2 하루 타임라인

시스템과의 접점은 **정기 2회 + 주말 심화**. 낮 시간에는 상호작용하지 않는다.

| 시간대 | 주된 활동 | 시스템 접점 | 비고 |
|---|---|---|---|
| 아침 (기상~출근) | 오늘 할 일 정리 | 브리핑 수신·플랜 확정 | 하루 시작 트리거 |
| 오전~오후 | 업무·공부 집중 | **없음** (방해 금지) | 이 시간대에 시스템이 말을 걸면 안 됨 |
| 저녁 (저녁~취침 전) | 하루 마무리 | 회고·승인 인박스 처리 | 제안·엔티티 초안 등 큰 변경은 이 시간에 일괄 처리 |
| 주말 | 여유 블록 | 심화 세션 (학습·상담·주간 리뷰) | 드물지만 깊이 있는 사용 |

→ "**하루 2번 의식적 접점 + 주말 심화**"가 기본 리듬. 이 리듬을 깨는 알림·nudge는 최소화한다.

## 2.3 사용 환경

### 하드웨어 구성

- **상시 가동 허브**: 원격으로 항상 켜져 있는 기기 1대. 백그라운드 프로세스·스케줄·에이전트 상주
- **작업용 로컬 기기**: 주 사용 컴퓨터. 허브와 네트워크로 연결되어 클라이언트 역할
- **휴대 기기**: 메시징·가벼운 입력

### 운영 전제

- 허브는 24시간 가동. 사용자 부재 중에도 외부 시그널 감지·스케줄 실행 가능
- 개인정보가 들어 있으므로 **로컬 우선**이 기본 원칙
- 데이터량은 하루 ~100건 이하 입력 예상 (장기 보관)

## 2.4 입력 수단 3종

사용자가 시스템에 정보를 넣는 주 경로.

1. **터미널 입력**: 즉시 메모·질의·조회. 가장 친숙한 환경.
2. **채팅 앱 양방향**: 이동 중·다른 작업 중 접근. 대화형 플랜 확정에 주로 사용.
3. **마크다운 파일 직접 타이핑**: 별도 텍스트 편집기·노트 앱에서 파일로 기록. 파일 변경이 자동 감지되어 시스템이 수집.

**명시적 제외**:
- 음성 녹음 입력 — MVP 범위에서 빼기로 결정. 텍스트 중심으로 좁힘.
- 웹·브라우저 UI — MVP에 포함 안 함.

---

## 2.5 출력 채널 4종

시스템이 사용자에게 응답·결과를 전달하는 경로. 각 채널은 역할이 다르다.

| 채널 | 주 역할 | 언제 쓰이나 |
|---|---|---|
| 채팅 앱 양방향 | 대화형 플랜 확정·넛지 | 아침 브리핑, 플랜 확정, 중요 알림 |
| 외부 태스크 시스템 | 확정된 플랜의 영속 저장 | 사용자가 승인한 태스크를 외부 도구에 기록 |
| 터미널 | 즉시 조회·검색 | 사용자가 먼저 물을 때만 |
| 노트 파일·대시보드 | 장기 보존·조망 | 회고·리뷰·워크플로 세션의 결과물 |

→ 같은 정보라도 **어디로 보내는가**가 다르다. 아침 브리핑은 채팅 앱으로, 회고 결과는 노트 파일로, 즉시 질문은 터미널로.

## 2.6 능동 개입 허용 범위

시스템이 사용자 요청 없이 **먼저** 말을 걸어도 되는 상황. **엄격히 제한**.

### 허용
- **정해진 시간**: 아침·저녁 스케줄된 브리핑·회고
- **외부 시그널 감지 시**: 사용자가 연동해둔 시스템에 새 이벤트·변화가 있을 때

### 금지
- **부재·누락 감지**: "운동 3일 안 함", "연락 끊김" 같은 nudge는 **하지 않는다**
- **아무때나 알림**: 위 2가지 외의 능동 개입은 금지

→ 이 원칙은 침입성 제어의 핵심. 사용자는 **능동형 메모리**(먼저 "기록할까?" 제안)는 원하지만 **잔소리형 감지**는 원하지 않는다. 둘은 다르다.

## 2.7 사용 맥락 시나리오 5개

구체적인 사용 장면. 요구사항 해석의 검증 기준.

### S1. 면접 직후 기록

"방금 끝난 원격 면접. 15분 안에 후기를 남기고 싶음. 면접관 이름과 인상을 간단히 적으면 다음번 2차 면접 때 자동으로 끌려와야 함."

### S2. 이동 중 아이디어

"걷다가 떠오른 한 줄 생각. 폰에서 잠깐 꺼내 기록. 나중에 집에서 정리할 때 이 한 줄이 관련 맥락과 연결돼야 함."

### S3. 아침 오늘 할 일

"기상 직후 '오늘 뭐 해야 하지?' — 어제 못 한 것 + 오늘 예정 + 최근 외부 변화 기반 제안이 한 번에 오고, 몇 번 주고받아서 확정."

### S4. 과거 맥락 소환

"지난 달 Anthropic 관련해서 뭐가 있었지?" — 관련 모든 기록(면접 후기·메일 요약·관련 인물)이 한 번에 나열."

### S5. 학습 세션 이어받기

"ML 공부 세션 7회차. 지난번에 어디까지 했고 오늘 뭐 할지부터 시작. 내가 '지난번 뭐 했지' 말 안 해도 됨."

→ 위 5개는 요구사항·기능이 실제로 이 순간들을 매끄럽게 처리하는지 **검증 기준**으로 사용.
